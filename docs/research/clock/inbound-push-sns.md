# Clock Inbound PUSH/SNS

## Scope

Protocol and implementation notes for Clock PMS+ inbound booking synchronization through Amazon SNS.

## Verified implementation claims

- Clock PUSH sends Amazon SNS envelopes to an HTTP/S endpoint.
- Clock documents optional HTTP Basic authentication by embedding credentials in the endpoint URL.
- Clock `Notification.Message` is a JSON-formatted string containing identifiers such as `booking_id`.
- Documented booking subjects are `booking_new`, `booking_update`, `booking_expected`, `booking_checked_in`, `booking_checked_out`, `booking_canceled`, `booking_no_show`, and `booking_guests_update`.
- Clock documents up to three PUSH retries with a 20-second delay when the endpoint is unreachable.
- SNS `MessageId` remains stable across a retry of the same message.
- SNS signatures must be verified before confirmation or notification processing. Signature version 1 uses SHA-1 and version 2 uses SHA-256.
- Subscription confirmation must follow `SubscribeURL` only after signature and URL validation.

## Operational requirements

- Public HTTPS endpoint: `/wp-json/must-hotel-booking/v1/clock/webhook`.
- The Clock account must use the exact URL generated by the deployed WordPress site.
- Optional Basic authentication is valid only when both username and password are configured.
- Clock provider mode, inbound route registration, booking fetch access, and required room mappings must be available.
- Web server, CDN, firewall, maintenance mode, and security plugins must allow SNS POST bodies and the `Authorization` header when Basic auth is used.

## Current implementation audit

The route is registered by `ClockInboundSyncController` and bootstrapped by `Plugin`. It preserves the raw request body for signature verification, parses SNS envelopes, validates SNS certificate and subscription URLs, verifies signature versions 1 and 2, confirms subscriptions, decodes nested notification JSON, logs sanitized summaries, and delegates booking updates to `ClockInboundSyncService`.

Confirmed defects fixed on 2026-06-18:

- A transient Clock booking fetch failure could still return `200`, mark the SNS `MessageId` successful, and permanently suppress a later retry even though no reservation state changed.
- Documented status-specific subjects were not used as a safe fallback when the booking fetch failed.
- Half-configured Basic authentication acted as a wildcard for the missing username or password.
- Malformed nested SNS `Message` JSON was not rejected explicitly.
- Subscription confirmation and notification deduplication were not serialized, allowing concurrent duplicate processing.
- Repeated successful subscription confirmations followed `SubscribeURL` again.

Current behavior:

- Status-specific Clock subjects can apply safe local status fallbacks and still queue a detail refresh.
- `booking_update`, `booking_new`, and other detail-only events return retryable `503` when Clock details cannot be fetched.
- Missing mapping/upsert failures are retryable, queue a `booking_upsert` reconciliation job, and are not logged as successful processing.
- Successful duplicates are no-ops; failed first attempts remain retryable.
- Unsupported non-booking events are logged as unsupported and acknowledged without claiming a booking update.

## Configuration and reachability evidence

- Local plugin version inspected: `0.4.81`.
- Local provider mode: Clock.
- Local Clock environment: sandbox.
- Local booking fetch path: `/bookings/{booking_id}`.
- Local automatic Clock sync: enabled.
- Local Clock API credentials: configured, values not recorded.
- Local Clock mappings: 4 accommodation, 81 physical-room, and 4 rate-plan mappings.
- Optional Basic authentication: not configured.
- Legacy token/HMAC secret: not configured; not required for official SNS.
- Effective local callback: `https://headwear-cranium-daydream.ngrok-free.dev/wp-json/must-hotel-booking/v1/clock/webhook`.
- The effective local callback returned a controlled `403` for unsigned `{}` without redirecting.
- Expected production callback `https://new.empirebeachresort.al/wp-json/must-hotel-booking/v1/clock/webhook` timed out during connection from the audit environment.

The stored local callback therefore does not match the expected production URL, and the deployed production plugin/version and Clock account PUSH configuration could not be verified.

## Remaining protocol hardening

AWS recommends rejecting unexpected `TopicArn` values and validating the signing certificate chain/issuer. The implementation restricts certificate and subscription URLs to HTTPS Amazon SNS hosts and relies on WordPress HTTPS validation, but no configured topic allow-list is currently available. This should be treated as follow-up hardening before a production-ready claim.
